Collaborative filtering- plays, skips, saves, watch time
Content-based filtering-- audio features (tempo, mood, energy, valence, danceability, acousticness, insrtumentalness, loudness, key), lyrics, transcripts, tags, visuals

Recommendation Data
├── Behavioral (Event Streams)
│   ├── Play / Skip / Save / Repeat  →  Boolean, Enum, Integer, Timestamp
│   └── Watch time / Session length  →  Float
│
├── User Profile
│   ├── Demographics               →  Categorical
│   ├── Context (time, device)     →  Categorical + Temporal
│   └── Latent taste vector        →  Dense Float Array (40–200d)
│
├── Audio Features (Spotify-specific)
│   ├── Scalar features            →  Float (0.0–1.0 range)
│   └── Deep audio embeddings      →  Dense Float Array (~128d)
│
├── Item Metadata
│   ├── IDs, dates, flags          →  String, Date, Boolean
│   └── Genre / language tags      →  Categorical / List of Strings
│
├── Text / NLP
│   ├── Raw text                   →  String (never used directly)
│   └── Embeddings                 →  Dense Float Array (100–768d)
│
└── Inferred / Derived
    ├── Mood scores                →  Float (often 0.0–1.0 per mood class)
    └── Popularity / freshness     →  Float + Timestamp

Spotify has no star ratings. It learns from implicit signals:
- Stream >30 seconds → positive signal
- Hard skip <30 seconds → negative signal
- Save / add-to-playlist → strong positive
- Repeat plays → very strong positive

Claude's best recommendation:
- genre + mood + energy + valence
- skip: tempo_bpm because redundant with energy
- skip: danceability and acousticness complicate a similarity calculation

I generally agree that tempo_bpm, energy, and danceability are largely the same but I think acousticness
adds some specific nuance.

Algorithm Recipe: genre + mood + energy + valence + acousticness

---

Scoring Rule (genre, mood, energy, acousticness — Mood-Relative Weights)
--------------------------------------------------------------
score(s, p) = w_genre        * match(s.genre, p.genre)
            + w_mood         * match(s.mood, p.mood)
            + w_energy       * (1 - |s.energy - p.energy|)
            + w_acousticness * match(round(s.acousticness), p.likes_acoustic)

Where:
  match(a, b)       = 1  if a == b
                      0  otherwise
  round(x)          = 1  if x >= 0.5  (acoustic)
                      0  if x <  0.5  (not acoustic)
  p.likes_acoustic  = 1 or 0  (user binary preference)

Weights (mood = 1.0 as base; all others relative to it):
  w_genre        = 2.0   ← strongest preference signal
  w_mood         = 1.0   ← base weight
  w_energy       = 0.8   ← most discriminating numeric feature
  w_acousticness = 0.5   ← binary texture check

Max possible score = 2.0 + 1.0 + 0.8 + 0.5 = 4.3
  (when every feature is a perfect match)

To get a normalized score in [0, 1]:
  score_normalized(s, p) = score(s, p) / 3.5

User profile p is computed from liked songs:
  p.genre          = most_common(liked_songs.genre)
  p.mood           = most_common(liked_songs.mood)
  p.energy         = mean(liked_songs.energy)
  p.likes_acoustic = 1 if mean(liked_songs.acousticness) >= 0.5 else 0

---

Ranking Rule
-------------
After computing score(s, p) for every unheard song s:

1. Sort all candidate songs by score descending.
2. Return top-k (e.g. k=3 for this dataset).
3. Tiebreaker: if two songs have scores within 0.01 of each other,
   prefer the one with energy closer to p.energy (most context-sensitive feature).

Final ranking formula:
  rank = sorted by score DESC, then by |s.energy - p.energy| ASC on ties
